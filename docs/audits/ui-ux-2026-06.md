## 1. Executive Summary

Irogen's UI/UX is in **good shape overall** — DaisyUI adoption is broad, theming is consistent, the layout-polish work earlier this session unified spacing and width tokens, and most interactions follow sensible patterns.

However, there are **5 high-impact issues** that should be fixed before shipping to non-technical users:

| # | Issue | Severity | Effort |
|---|-------|----------|--------|
| 1 | Focus styles explicitly removed in `src/index.css` (lines 30-46) | 🔴 P0 | XS |
| 2 | 13 unused UI primitive files (~1.5k LOC of dead code) | 🟠 P1 | S |
| 3 | All scrollbars hidden globally (`*::-webkit-scrollbar { display: none }`) | 🟠 P1 | S |
| 4 | No `sr-only` labels on icon-only buttons | 🟠 P1 | S |
| 5 | `index.css` focus-removal block applies to native + DaisyUI inputs | 🔴 P0 | XS |

The rest of this report is organized by **theme** with concrete file:line references.

## 3. Critical Findings 🔴

### F1. Focus styles globally suppressed
**File:** `src/index.css:30-46`
```css
input:focus,
textarea:focus {
  outline: none !important;
}
.input:hover, .input:focus, .input:focus-visible,
.textarea:hover, .textarea:focus, .textarea:focus-visible,
.select:hover, .select:focus, .select:focus-visible {
  outline: none !important;
  box-shadow: none !important;
}
```
**Impact:** Keyboard users have **no visible focus indicator** on:
- Chat input (the most-used element)
- Ticket input on Home
- Settings inputs (theme/language/ticket)
- Permission mode selectors
- All `<input>` and `<textarea>` elements

This is a WCAG 2.4.7 violation (Focus Visible) and the **#1 blocker for accessibility**.

**Fix:** Replace with DaisyUI's native focus ring (don't override), or use `focus-visible:` Tailwind utilities instead of removing outline. Suggestion:
```css
.input:focus-visible,
.textarea:focus-visible {
  outline: 2px solid hsl(var(--p));
  outline-offset: 2px;
}
```

### F2. Focus ring missing on most interactive elements
**Files:** 17+ components use `<button>`/`<div role="button">` without any `focus-visible:` classes
- Only 4 files use `focus:` / `focus-visible:`: `dialog.tsx`, `SettingsView.tsx`, `AgentPanel.tsx`, `TcpForwardingModal.tsx`
- **Session sidebar rows** (most-clicked elements) have no focus ring
- **Session list rows** in `SessionsView` have no focus ring
- **Ticket cards** on Home have no focus ring
- **Recent sessions** list rows (was removed but if re-added) would need it

**Fix:** Add `focus-visible:ring-2 focus-visible:ring-primary/50 focus-visible:outline-none` as a default utility class on all `card card-bordered` clickable elements, OR define a reusable `.focus-ring` CSS class in `index.css`.

**Fix:** Delete them. If any need reviving, git history has them.

### F4. No visible scrollbars anywhere
**File:** `src/index.css:14-19`
```css
* { scrollbar-width: none; }
*::-webkit-scrollbar { display: none; }
```
**Impact:**
- Desktop users on long session lists / settings panels can't tell they can scroll
- On Windows with mouse, the absence of scrollbar = no obvious "more content here" signal
- Power users with scroll wheel still work, but **discoverability suffers**

**Fix:** Use DaisyUI's `scrollbar-thin` utility or define a custom thin scrollbar:
```css
*::-webkit-scrollbar { width: 8px; height: 8px; }
*::-webkit-scrollbar-thumb {
  background: hsl(var(--bc) / 0.2);
  border-radius: 4px;
}
*::-webkit-scrollbar-thumb:hover {
  background: hsl(var(--bc) / 0.4);
}
```

### F5. No screen-reader-only labels on icon buttons
**Pattern:** Many buttons use only `title="..."` with no `aria-label`. Examples:
- `SessionSidebar.tsx` — collapse toggle button
- `SessionSidebar.tsx` — FiStopCircle / FiX session actions
- `HomeView.tsx` — start new session `+` button
- `ChatView.tsx` — copy/quote/resend buttons in MessageBubble
- `ChatHeader.tsx` — toggle buttons (file browser, git panel, permission switcher)

`title` alone is not sufficient — screen readers may or may not announce it, and it's not localized.

**Fix:** Add `aria-label="..."` to every icon-only button. Bonus: add `sr-only` span for any button that uses only an icon with no visible text.

### F6. i18n coverage is incomplete
- Only `AppLayout.tsx` uses `i18nStore.t()` directly
- `rg 't\("' src/components/` — 9 components use the inline `t("...")` helper
- Most strings inside `MessageBubble`, `ChatView`, `ChatInput`, `SessionSidebar` are **hardcoded English**:
  - `MessageBubble.tsx`: "Stop", "Close", "Copy", "Quote and reply", "Resend message"
  - `ChatInput.tsx`: "Send", "Stop"
  - `NewSessionModal.tsx`: "Loading...", "Connected to host", "Connection failed"

**Impact:** The `zh-CN` dictionary in `i18nStore.ts:301-336` exists but is mostly unused. Non-English users get a mixed-language UI.

**Fix:** Either commit to full i18n or remove the unused Chinese dictionary (current state is the worst of both worlds).

### F7. Inline empty states duplicate logic
Three views each define their own empty state JSX:
- `SessionsView.tsx:138-152` (sessions empty)
- `HomeView.tsx:186-193` (was Recent Sessions empty — still inline fallback)
- `SessionSidebar.tsx:551-568` (sidebar empty)
- `FileBrowserView.tsx` (file browser empty)

`EnhancedEmptyState.tsx` exists specifically to consolidate these but is **unused**.

**Fix:** Either delete `EnhancedEmptyState.tsx` (it's dead) or refactor the 4 inline empty states to use it. Recommend consolidation — would reduce ~80 LOC.

### F8. Two parallel toast implementations
- `src/components/ui/Toast.tsx` — defines `ToastContainer` and exports (unused)
- `src/components/ToastContainer.tsx` — actual implementation wired in `App.tsx`

**Fix:** Delete `ui/Toast.tsx` or merge into `ToastContainer.tsx`.

## 5. Medium-Priority Findings 🟡

### F9. User messages lack avatar
**File:** `src/components/ui/MessageBubble.tsx:484-505` (`UserMessage`)
- User messages render with no avatar, only timestamp + bubble
- Assistant messages have circular avatar + status badge
- Asymmetric — user can't tell at a glance if they're the one who spoke

**Fix:** Add a user initial avatar (consistent with sidebar), or omit timestamp when there's no avatar to pair it with. Could use `user?.name` from settingsStore for personalization.

### F10. No session drag-to-reorder
Sessions in `SessionSidebar.tsx` are sorted by `startedAt`. Users can't:
- Pin a session to top
- Reorder by custom priority
- Drag a session to a different project group

Common pattern in chat apps (Slack, Discord). Not blocking but expected.

### F11. No right-click context menu on sessions
Right-click on a session does nothing. Users expect:
- Rename session
- Move to project
- Copy session ID
- Export transcript

**Fix:** Add `onContextMenu` → custom `ContextMenu` component (would need a primitive — `CyberComponents.tsx` is dead code that could be revived or new primitive built).

### F12. ChatInput always-visible send button can be obscured
`src/components/ui/ChatInput.tsx` — send button is positioned absolute-right inside the input. On narrow viewports (320px) it can overlap text. Also the textarea has no `min-rows`, so 1-row input + button looks cramped.

### F13. Permission cards dismiss after decision without animation
`src/components/ui/PermissionCard.tsx` — clicking Accept/Reject removes the card instantly. Should fade-out + slide-right for ~150ms to confirm action.

### F14. No keyboard shortcut to jump between sessions
Sidebar sessions are navigable via Tab but no `Cmd+1..9` to jump to nth recent session. Common power-user feature.

### F15. Status indicators inconsistent
- `Local Daemon` shows "Running" with green dot (good)
- `Connected Hosts` shows "Connected" badge (good)
- `Reconnecting` state in `SessionSidebar` shows neutral dot — should pulse or use warning color
- No "Connecting…" state for tickets mid-handshake

### F16. No bulk actions on sessions
SessionsView shows 5+ sessions with individual Resume/Stop buttons. Can't:
- Stop all active sessions
- Archive all inactive sessions
- Bulk delete

## 6. Low-Priority Findings 🟢

### F17. Animation count is high without visual variety
- `EnhancedMessageComponents.tsx` has 11 `animate-*` classes
- `AppLayout.tsx` has 6
- `TcpForwardingModal.tsx` has 4

Several animations are duplicates of each other (multiple `animate-fade-in` on the same view). Consider consolidating into utility classes in `chat-tokens.css`.

### F18. `Lucide` icon library partially used
`ToastContainer.tsx` uses `lucide-solid` while 26 files use `solid-icons/fi`. Inconsistent. Recommend standardizing on `solid-icons/fi` (already dominant).

### F19. `MessageListView.tsx` exports `ChatEmptyState`
`MessageListView` is imported by ChatView, but ChatView defines its own empty state inline. `ChatEmptyState` is exported from `chat/index.ts` but unused. **Dead export.**

### F20. No `prefers-color-scheme` media queries
App defaults to `light` theme. Users with dark-mode OS preference get light theme initially. Either:
- Auto-detect via `prefers-color-scheme` and default to it
- Add a first-run prompt: "Irogen detected your dark OS — switch to Abyss/Synthwave?"

### F21. Settings lacks reset/import/export
`SettingsView` has Theme + Language but no:
- Export settings to JSON
- Import settings
- Reset to defaults button

### F22. Sidebar unread badge relies on a single counter
`SessionSidebar` shows `totalUnread()` badge on Sessions header. Doesn't show per-session unread count on collapsed rows. If you have 5 sessions with unread, you only see one badge.

### F23. Toast dismiss button missing
`Toast.tsx:189` has `ToastContainer` (dead) but the live `ToastContainer.tsx` renders toasts without explicit close buttons. Auto-dismiss after duration only. Power users want to dismiss manually.

## 8. Cross-Cutting Observations

### What's working really well
- **Zed-inspired design language** is consistent: hard lines, no shadows, no gradients in most surfaces
- **Theme tokens** are deeply integrated — flipping themes in `ThemeSwitcher` cascades correctly through every component
- **Empty states are intentional** with icon + title + description + optional CTA — matches LobeHub's quality bar
- **Animations are subtle** — `transition-colors duration-150` is the dominant pattern, never flashy
- **Auto-scroll to bottom** in chat works correctly with streaming messages

### What needs discipline
- **350+ i18n keys defined** but used inconsistently — pick a direction
- **Components are mostly small** (sessions view 250 lines, settings view 200 lines) — easy to refactor
- **ChatView is the elephant** at 2012 lines — should be split, but no need to touch it now

### Risks if left unfixed
- **Keyboard-only / screen-reader users** are effectively excluded (F1, F2, F5)
- **Long-running maintenance** will suffer from dead-code rot (F3)
- **Visual regression in scrollable views** is invisible until someone reports it (F4)
